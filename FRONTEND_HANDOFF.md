# NORLAB frontend — полный handoff для переноса и подключения backend

Дата фиксации: 4 июля 2026 года.

Этот документ описывает текущее состояние интерфейса «Фабрика гипотез», расположение экранов и компонентов, реализованное поведение, источники mock-данных и безопасный порядок подключения backend. Внешний вид считается согласованным: при интеграции следует менять слой данных, а не перестраивать UI.

## 1. Кратко о продукте

NORLAB — интерфейс для научного сотрудника лаборатории или R&D-инженера. Система принимает brief, ограничения и мультимодальные источники, проводит исследовательский прогон, формирует и ранжирует гипотезы, показывает доказательства и позволяет собрать план эксперимента.

Предметная область зафиксирована:

- отвальные хвосты и шлаки;
- обогащение и металлургический передел;
- таблицы с историей экспериментов;
- PDF/текстовые отчёты;
- изображения схем цепей аппаратов;
- обоснование через физику, химию, термодинамику и механику процесса;
- промышленный синтез исключён.

Ограничения пользователь вводит обычным связным текстом. На сайте нет поля «промпт для генерации» и нет отдельного экрана «качество импорта».

## 5. Глобальная оболочка

Файл: `src/app/AppShell.tsx`.

Реализовано:

- логотип NORLAB возвращает на `/projects`;
- переключатель показывает все три проекта и сохраняет текущий раздел при смене проекта;
- навигация проекта: «Рабочая область», «Исследование», «Гипотезы», «Эксперименты»;
- RU/EN/中文 меняют интерфейс без смены URL;
- настройки открываются центральным модальным окном;
- стартовая синяя заставка появляется после полной загрузки страницы, плавно скрывается через 1550 мс и не повторяется при переходах внутри SPA.

Backend должен заменить список проектов и профиль пользователя, не меняя структуру шапки.

## 6. Портфель проектов

Файл: `src/features/projects/ProjectsPage.tsx`.

Реализовано:

- поиск по названию на клиенте;
- фильтры «Все», «Активные», «Требуют внимания»;
- три проекта с разными `projectId`, контекстами и документами;

Сейчас `submitProject(data)` выводит `FormData` в console и затем открывает mock-проект. Его нужно заменить на `api.createProject(...)` и перейти по `response.id`.

## 7. Рабочая область

Файл: `src/features/workspace/WorkspacePage.tsx`.

### Обзор

- readiness 82%;
- список ключевых источников;
- brief с выделенной проблемой, целью и критериями;
- редактирование brief в центральном окне;
- следующий шаг и четыре предупреждения;
- предупреждения раскрываются рядом с триггером по hover/focus;
- блок похожих кейсов памяти института.

### Данные

- загрузка дополнительных файлов;
- локальный список загруженных `File`;
- статусы готовности;
- открытие сведений о каждом файле.

### Предпросмотр файлов

По умолчанию показывается только экран «О файле». Содержимое не читается и не отображается до нажатия «Предпросмотр». Повторное нажатие возвращает сведения о файле.

Поддержано:

- PDF — встроенный browser viewer для реального `File`, демонстрационный постраничный viewer для mock PDF;
- DOCX — извлечение текста через mammoth, включая русскую кириллицу;
- XLSX/CSV — табличный viewer;
- изображения — фактическое изображение;
- TXT/MD/JSON/XML/HTML/LOG/RTF — текст с UTF-8 и fallback Windows-1251;
- неизвестный бинарный формат — понятное сообщение и возможность скачать, без вывода бинарного мусора.

Для backend предпочтительно отдавать метаданные отдельно, а содержимое/preview по отдельным URL. Не вставлять base64 документа в общий `workspace` response.

### Граф связей

Граф показывает происхождение решения, а не абстрактные связи:

```text
файл/схема → извлечённый факт → гипотеза → эксперимент и риск
```

У узла есть тип, уверенность, описание и ссылка на исходный файл/страницу. У ребра есть relation, weight и текстовое объяснение. Текущая реализация интерактивна: выбор узла меняет информационную панель.

## 8. Исследовательский прогон

Файл: `src/features/research/ResearchPage.tsx`.

Реализовано:

- этапы Memory → Generation → Deduplication → Gates → Critique → Final;
- текущий этап и статус;
- прошедшее время и ETA;
- лента событий;
- восстановление с последнего подтверждённого этапа;
- блокирующее уточнение с ответом обычным текстом и комментарием;
- воронка 32 → 24 → 12 → 6 → 3;
- skeleton будущих результатов;
- кандидаты и deep-link в гипотезы;
- параметры ранжирования;
- четыре веса автоматически перераспределяются и всегда дают 100%;
- формула рейтинга и текстовые доменные ограничения.

Для живого backend рекомендуется SSE. Состояние прогона должно также читаться обычным GET, чтобы F5 и временный обрыв SSE не ломали страницу.

## 9. Гипотезы

Файл: `src/features/hypotheses/HypothesesPage.tsx`.

Реализовано:

- карточки с главной идеей, физико-химическим механизмом, ключевым условием, рейтингом, KPI, риском и новизной;
- шкалы науки, инженерии, механизма и тестируемости;
- легенда шкал и формула интегрального рейтинга;
- семантические цвета новизны и риска;
- локальная сортировка и фильтрация;
- выбор гипотез для сравнения;
- сравнительная матрица;
- создание экспертной гипотезы;
- центральная подробная карточка с вкладками Overview/Evidence/Novelty/Risks/Uncertainty/History;
- Engineering Gates;
- экспертный verdict, причина и свободный комментарий;
- evidence со ссылками на источник;
- встроенный постраничный reader источника;
- deep-link через `?hypothesis=...&panel=...`;
- переход к созданию эксперимента.

Backend обязан сохранять traceability: evidence содержит `source_file_id`, page/paragraph, quote и claim. Ссылка «Открыть источник» не должна формироваться из текста имени файла.

## 13. Важные UX-инварианты

1. Главный `/projects` не увеличивать: его масштаб отдельно зафиксирован в CSS.
2. Внутренние экраны, модальные окна, списки и текст намеренно крупнее главной страницы.
3. Гипотеза открывается в центре, не в боковой панели.
4. Ограничения и уточнения вводятся связным текстом.
5. Содержимое файла скрыто по умолчанию.
6. «Предпросмотр» — toggle: первый click открывает, второй возвращает «О файле».
7. Источники читаются внутри сайта, если backend способен предоставить preview.
8. Сумма весов ранжирования всегда ровно 100%.
9. ETA и восстановление прогона должны оставаться видимыми.
10. Roadmap обязан показывать зависимости и критический путь.
11. Язык интерфейса не должен менять route или `projectId`.
12. Для новых/неизвестных данных нужен skeleton/error/empty state, а не пустой белый блок.

## 15. Перенос на другой ПК

### Через подготовленный архив

1. Распаковать архив так, чтобы рядом находились `frontend/` и `docker-compose.yml`.
2. Установить Docker Desktop либо Node.js 22 LTS.
3. Не переносить старый `node_modules`; выполнить `npm ci` на новом ПК.

### Запуск через Node

```bash
cd frontend
copy .env.example .env.local
npm ci
npm run dev
```

Открыть `http://127.0.0.1:4173`.

### Запуск frontend через Docker

```bash
docker compose up --build
```

Открыть `http://127.0.0.1:4173`.

### Проверки после переноса

```bash
npm run lint
npm test -- --run
npm run build
npm audit --audit-level=high
npx playwright install chromium
npm run test:e2e -- --project=chromium
```

## 16. Definition of Done для подключения backend

- все пять маршрутов читают реальные данные;
- F5 сохраняет projectId, tab и открытый deep-link гипотезы;
- create project отправляет multipart и переходит на ID созданного проекта;
- загрузка файлов показывает lifecycle uploaded → parsing → ready/warning/failed;
- просмотр и скачивание работают по file ID;
- прогон восстанавливается после reconnect;
- уточнения и ranking profile сохраняются;
- гипотезы содержат evidence с traceability;
- feedback сохраняется;
- experiment revision не теряется;
- экспорт создаёт job и отдаёт download URL;
- есть loading/error/empty/partial states;
- lint, unit, e2e, production build и Docker проходят;
- пять ключевых скриншотов визуально совпадают с baseline.
